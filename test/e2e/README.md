# End-to-End Tests

The e2e test suite validates the ECR Secret Operator against real AWS infrastructure. It provisions a ROSA HCP cluster and ECR repository, installs the operator via OLM, and runs a shell-based test suite covering both `Secret` and `ArgoHelmRepoSecret` custom resources.

## How it works

The e2e pipeline uses two GitHub Actions workflows:

- **`e2e-trigger.yaml`** — a lightweight `issue_comment` handler that parses the `/e2e` command, checks CODEOWNER authorization, and dispatches the real workflow. It has no access to secrets and never checks out PR code.
- **`e2e.yaml`** — the `workflow_dispatch`-only workflow that does the actual work. Because it's triggered by dispatch (not directly by the comment event), it runs in a trusted context.

```
e2e-trigger.yaml (issue_comment)
      │
      │  gh workflow run e2e.yaml
      ▼
e2e.yaml (workflow_dispatch)
      │
resolve-pr          Look up PR head SHA and labels
      │
provision           Terraform apply: VPC, ROSA HCP cluster, ECR repo, IAM role
      │
test                Install operator via OLM, run test suite
      │
destroy             Terraform destroy (skipped in debug mode)
```

### Triggering

All commands are issued as PR comments or via workflow dispatch:

| PR Comment | Workflow Dispatch | Behavior |
|---|---|---|
| `/e2e` | command: `run` | Provision, test, and destroy infrastructure |
| `/e2e debug` | command: `debug` | Provision and test, but **preserve** infrastructure for investigation |
| `/e2e destroy` | command: `destroy` | Destroy infrastructure only (for cleanup after `/e2e debug` or a failed run) |
| `/e2e force-destroy` | command: `force-destroy` | Remove the ROSA cluster from Terraform state, then destroy remaining resources. Use when the cluster is stuck deleting or already gone |

Only users listed in `CODEOWNERS` are authorized to trigger e2e runs via comments. Workflow dispatch (Actions UI) bypasses the CODEOWNER check.

### Prerequisites

The e2e workflow depends on images built by the CI workflow (`ci.yaml`). CI runs automatically on every PR and pushes:

- `quay.io/rh-mobb/ecr-secret-operator:pr-<number>`
- `quay.io/rh-mobb/ecr-secret-operator-bundle:pr-<number>`

Wait for CI to complete before commenting `/e2e`.

### Authorization and environment gates

The `provision` and `destroy` jobs target the `e2e-test` GitHub Actions environment, which can be configured with protection rules (required reviewers, deployment branches, etc.) under **Settings > Environments**.

AWS credentials are obtained via GitHub OIDC federation — no static credentials are stored. The workflow assumes `arn:aws:iam::660250927410:role/ecr-secret-operator-e2e` and requires `id-token: write` permission.

ROSA cluster provisioning uses OCM service account credentials (`RHCS_CLIENT_ID` and `RHCS_CLIENT_SECRET`), stored as repository secrets.

## Infrastructure

All infrastructure is defined in Terraform under `test/e2e/terraform/`. Terraform state is stored in an S3 backend keyed by PR number (`pr-<number>/terraform.tfstate`), so multiple PRs can have independent clusters.

### Resources provisioned

| Resource | Purpose |
|---|---|
| VPC (single AZ) | Network for the ROSA cluster, kept minimal to reduce cost |
| ROSA HCP cluster | OpenShift cluster with 2x `m7i.xlarge` workers |
| Cluster admin password | Generated by `random_password`, passed to the test job |
| IAM OIDC role | Federated to the cluster's OIDC provider, grants `ecr:GetAuthorizationToken` |
| ECR repository | Target registry for the operator to authenticate against |
| VPC endpoint cleanup (`null_resource`) | Runs on destroy to clean up orphaned VPC endpoints and security groups that can block VPC deletion |

All AWS resources are tagged with `app-code`, `service-phase`, `owner`, `repo`, and `pr` for traceability.

### Force-destroy behavior

When `/e2e force-destroy` is used, the workflow:

1. Removes `module.rosa_hcp`, `aws_iam_role.operator_ecr`, and `aws_iam_role_policy.operator_ecr` from Terraform state (the IAM resources reference the cluster's OIDC endpoint, which becomes unresolvable once the cluster module is removed)
2. Deletes the IAM role and its inline policy directly via the AWS CLI
3. Runs `terraform destroy` on the remaining resources (VPC, ECR, `null_resource`)

### Preserving infrastructure

To keep infrastructure alive after a test run:

- Comment `/e2e debug` instead of `/e2e` — this skips the destroy job entirely
- Alternatively, add the `e2e-keep-infra` label to the PR before commenting `/e2e` — this also skips the destroy job

When you're done investigating, comment `/e2e destroy` to tear everything down.

## Test suite

The test suite is a shell script at `test/e2e/scripts/run-e2e.sh`. It uses `oc` to apply CR manifests (from `test/e2e/manifests/`) and validates the operator's behavior.

### Test cases

| # | Test | What it validates |
|---|---|---|
| 1 | Secret CR creates dockerconfigjson secret | Applying a `Secret` CR produces a `kubernetes.io/dockerconfigjson` secret with valid data, and the CR status reaches `Updated` with a `lastUpdatedTime` |
| 2 | ArgoHelmRepoSecret CR creates repo secret | Applying an `ArgoHelmRepoSecret` CR produces a secret with `username`, `password`, `url`, `type`, `enableOCI`, and `name` keys, the `argocd.argoproj.io/secret-type: repository` label, and `Updated` status |
| 3 | Delete and recreate Secret CR | Deleting and recreating a `Secret` CR produces a new generated secret and reaches `Updated` status |
| 4 | Reconciliation recreates deleted secret | Deleting the generated secret and bumping the CR's `frequency` field (to trigger a `GenerationChangedPredicate` reconcile) causes the operator to recreate the secret |
| 5 | Invalid region does not crash operator | A `Secret` CR with `region: xx-invalid-99` does not produce a secret and does not crash the operator pod |
| 6 | Non-existent registry fails gracefully | A `Secret` CR pointing to `000000000000.dkr.ecr...` does not crash the operator |

### CR templates

Templates in `test/e2e/manifests/` use `envsubst` for variable substitution:

| Variable | Source |
|---|---|
| `ECR_REGISTRY` | Terraform output `ecr_registry_url` |
| `TEST_NAMESPACE` | Hardcoded to `ecr-secret-operator` |
| `AWS_REGION` | Defaults to `us-east-2` |
| `ECR_REPO_NAME` | Defaults to `ecr-e2e-test-repo` |

## Directory layout

```
test/e2e/
├── manifests/
│   ├── argo-helm-cr.yaml.tpl           # ArgoHelmRepoSecret CR template
│   ├── secret-cr.yaml.tpl              # Secret CR template (valid)
│   └── secret-cr-invalid-region.yaml.tpl  # Secret CR template (invalid region)
├── scripts/
│   └── run-e2e.sh                      # Test runner
└── terraform/
    ├── main.tf                         # VPC, ROSA cluster, ECR, IAM
    ├── outputs.tf                      # Values passed to the test job
    ├── variables.tf                    # Configurable inputs
    └── versions.tf                     # Provider version constraints
```
